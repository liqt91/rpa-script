浏览器扩展元素采集与推荐方案重构方案

背景与问题

当前扩展的元素采集和推荐方案生成使用两套独立的数据源：

环节函数数据来源问题
采集页面层级buildElementPathcur.attributes、parent.children给 sidepanel 展示用
生成推荐方案generateLocators / _levelCandidates重新遍历 el.attributes、parent.children生成候选列表

两套遍历独立执行，即使都在同一个 click 事件里，也可能因为以下原因产生不一致：

  • Property vs Attribute：React/Vue 通过 IDL property 设置 href/value，attributes 集合中看不到，但推荐算法重新查询 DOM 时可能读到不同的值。
  • Shadow DOM 边界：parentElement 在 shadow boundary 处停止，path 断链，但推荐算法可能生成基于逻辑父元素的选择器。
  • 动态 DOM：采集后、生成前框架重渲染，导致 index、classes、attributes 变化。
  • iframe 跨域：buildElementPath 无法穿透跨域 iframe，但推荐算法如果通过其他方式访问到 iframe 内元素，会产生完全不对应的候选。

直接后果：用户点击推荐方案后，applyCandidateToUI 的反向匹配频繁失败——[href="..."] 找不到对应属性、:nth-of-type(1) 对应不上 node.index、[role="banner"] 在祖先上但 fallback 只检查目标元素。

目标

  1. 单源 truth：推荐方案的每个 segment 必须能精确映射到 buildElementPath 已采集的某个 path[i]，消除猜测式反向匹配。
  2. 双向一致性：手动勾选生成的选择器，与基于 path 生成的推荐方案，在相同语义下必须输出完全一致的语法。
  3. 可扩展：为二期接入 DrissionPage CDP snapshot 预留接口，不破坏前端闭环能力。

技术方案

3.1 核心思路：基于 Path 生成推荐

不再让 generateLocators 重新查询 DOM，而是接收 buildElementPath 产出的 path 数组，基于 path[i] 的已采集字段生成候选。

用户 Alt+Click
    │
    ▼
buildElementPath(element)  ──►  path[]
    │                              ├── tag, id, classes, attrs
    │                              ├── index, realIndex
    │                              ├── childrenTags, siblings
    │                              └── parentRef (预留)
    │
    ▼
generateLocatorsFromPath(path)  ◄──  不再查 DOM
    │
    ▼
candidates[] + pathMapping[]  ──►  sidepanel

path 节点结构保持不变：

{
  tag: 'a',
  id: '',
  classes: ['css-1xvhq7k'],
  attrs: { href: 'https://...', role: 'link' },
  index: 0,          // same-tag sibling index
  realIndex: 6,      // all-children index
  childrenTags: ['div', 'a', 'span', 'a', 'a'],
  siblings: [...]
}

3.2 Level 候选生成（基于 Node）

function _levelCandidatesFromNode(node) {
  const level = [];

  if (node.id) {
    level.push({ name: '#' + cssEscape(node.id), penalty: 0 });
  }

  node.classes.filter(isStableClass).forEach(c => {
    level.push({ name: '.' + cssEscape(c), penalty: 1 });
  });

  Object.entries(node.attrs).forEach(([k, v]) => {
    if (acceptedAttrs.has(k) || k.startsWith('data-')) {
      if (v && !attrValNeedsCssFallback(v) && v.length < 100) {
        level.push({ name: `[${cssEscape(k)}="${attrValueEscape(v)}"]`, penalty: 2 });
      }
    }
  });

  level.push({ name: node.tag, penalty: 5 });

  const sameTagCount = node.childrenTags?.filter(t => t === node.tag).length || 1;
  if (sameTagCount > 1) {
    level.push({ name: `${node.tag}:nth-of-type(${node.index + 1})`, penalty: 10 });
  }

  return level;
}

与当前 _levelCandidates(el) 的区别：

  • 不访问 el.attributes，直接读 node.attrs
  • 不访问 parent.children，直接读 node.childrenTags
  • index 直接使用已采集值，不再重新计算

3.3 结构路径生成（基于 Path 组合）

当前 buildStructuralCss 和 getElementCssPath 从目标元素逐层向上爬，每次重新查 DOM。重构后直接从 path 数组组合：

function buildStructuralCssFromPath(path) {
  const segs = [];
  for (let i = path.length - 1; i >= 0; i--) {
    const seg = buildSegFromNode(path[i]);
    segs.unshift(seg);
    const selector = segs.join(' > ');
    if (verifyCount(selector) === 1) return selector;
    if (seg.startsWith('#')) break;
  }
  return segs.join(' > ');
}

3.4 Sidepanel 映射简化

推荐方案附带 pathMapping，标明每个 segment 对应 path 的哪个 index：

candidates.push({
  syntax: 'css:[role="banner"] > div > div:nth-of-type(1) > div > nav > [href="..."]',
  pathMapping: [2, 3, 4, 5, 6, 7],  // 6 个 segment 对应 path 的 6 个层级
  family: 'css',
  score: 88,
  matchCount: 1
});

applyCandidateToUI 不再需要 parseSeg + segMatchesNode 的反向猜测，直接根据映射勾选：

function applyCandidateToUI(c) {
  if (!elementData?.path?.length) return;
  const path = elementData.path;

  // Reset
  pathEnabled = path.map(() => false);
  attrEnabled = {};
  path.forEach((_, i) => { attrEnabled[i] = {}; });

  if (c.pathMapping) {
    const segs = c.syntax.replace(/^css:/, '').split(/\s*>\s*|\s+/).filter(Boolean);
    segs.forEach((segStr, idx) => {
      const pathIdx = c.pathMapping[idx];
      if (pathIdx === undefined) return;
      const seg = parseSeg(segStr);
      applySegToNode(seg, pathIdx);
    });
  } else {
    // 兼容旧数据：回退到反向匹配
    legacyApplyCandidateToUI(c);
  }

  selectedPathIndex = pathEnabled.lastIndexOf(true);
  renderDomTree();
  renderProperties();
}

3.5 补充 IDL Property（前端兜底）

buildElementPath 中，在遍历 attributes 后，显式补充高频 IDL 属性，解决框架通过 property 设置值的问题：

// 补充框架通过 property 设置的值（attributes 中可能缺失）
if (cur.href && !attrs.href) attrs.href = cur.href;
if (cur.src && !attrs.src) attrs.src = cur.src;
if (cur.value !== undefined && !attrs.value) attrs.value = String(cur.value);
if (cur.checked !== undefined && !attrs.checked) attrs.checked = String(cur.checked);

架构边界

前端闭环（MVP）

扩展在离线状态下仍能完整工作：

  • buildElementPath + generateLocatorsFromPath 全部在 content script 中同步执行
  • sidepanel 基于 path 渲染和映射
  • 校验通过 document.querySelectorAll 在真实 DOM 上执行

后端增强（二期）

DrissionPage CDP snapshot 作为可选增强，不破坏前端闭环：

风险与缓解

风险影响缓解措施
采集后 DOM 变化path 中的 index 与实际 DOM 不一致，选择器校验失败保留 sidepanel 校验按钮作为最终兜底；校验失败时提示用户重新捕获
path 截断（>15 层）深层元素缺少足够祖先，组合不出唯一选择器放宽截断限制到 25；截断时 fallback 到结构路径（div:nth-of-type(3) > a）
Shadow DOM 不完整path 在 shadow boundary 断链，祖先缺失一期接受此限制，二期通过 CDP snapshot 补充
childrenTags 只存 tag 名生成 ancestor 候选时缺少兄弟的 id/class 信息一期足够（finder 组合只依赖 tag/index）；如需更精细的 ancestor 锚点，二期扩展 childrenTags 为 childrenMeta
