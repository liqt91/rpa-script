# 工作日志 — UIAutomation 桌面控件集成

**日期**: 2026-07-15（续）
**主题**: 为 RPA 平台引入第二套桌面操作方案（UIAutomation），与 Win32 API 方案并行

## 一、做了什么

### 1. Win32 指令重命名（加 Win32 后缀）

原有的 12 个桌面操作指令全部复制并改名，新旧共存：

| 旧名 (保留) | 新名 (Win32) | 说明 |
|---|---|---|
| `findWindow` | `findWindowWin32` | 按标题查窗口 |
| `clickControl` | `clickControlWin32` | 点击控件 |
| `inputControl` | `inputControlWin32` | 输入文本 |
| `pickFromPath` | `pickFromPathWin32` | 从元素库取控件 |
| `clickMenu` | `clickMenuWin32` | 点击菜单 |
| `closeWindow` | `closeWindowWin32` | 关闭窗口 |
| `openApp` | `openAppWin32` | 打开软件 |
| `wait` | `waitWin32` | 等待 |
| `sendKey` | `sendKeyWin32` | 模拟按键 |
| `findChild` | `findChildWin32` | 查找子控件 |
| `findSibling` | `findSiblingWin32` | 查找兄弟控件 |
| `findParent` | `findParentWin32` | 查找父控件 |

每个指令在 `commands/` 下有 JSON 定义，在 `desktop_commands/` 下有 Python handler。标签统一加 `(Win32)` 后缀，分类不变（紫色图标）。

### 2. 创建 UIAutomation 基础设施

**`_uia.py`** — UIA 工具库（~260 行）
- 封装 `uiautomation` 库，懒加载
- 核心函数：`find_window_by_title` / `find_child_by_name` / `click_element` / `set_text` / `pick_from_path`
- 控件字典格式统一：`{name, class_name, control_type, automation_id, rect, _uia_ctrl}`

### 3. 创建 UIA 版核心指令（4 个）

| 指令 | cmd | 功能 |
|------|-----|------|
| 查找窗口 | `findWindowUia` | UIA 模糊/精确查找顶层窗口 |
| 按元素库取控件 | `pickElementUia` | 从 DB 读 UIA 元素路径 → 逐层定位 |
| 点击控件 | `clickElementUia` | UIA InvokePattern 或 Click() |
| 控件输入 | `inputElementUia` | ValuePattern.SetValue → 降级 SendKeys |

标签加 `(UIA)` 后缀，绿色图标，分类 `桌面操作(UIA)`。

### 4. 更新数据模型

- `element_kind` 枚举增加 `"uia"` 值（`schemas.py`）
- 元素库前端 `ElementLibraryTab` 新增绿色 UIA 标签显示
- UIA 元素详情面板：显示名称、控件类型、类名、AutomationId、层级路径

### 5. 创建 UIA 控件拾取器

**`scripts/picker_uia.py`** — 混合方案：
- **高亮**: Win32 `WindowFromPoint` + XOR 覆盖窗口（亚毫秒级，不依赖 UIA）
- **捕获**: Alt+左键时调用一次 `uiautomation.ControlFromPoint` + `GetParentControl` 链回溯
- 输出路径结构与 Win32 picker 一致：`{name, class_name, control_type, automation_id, rect, path[]}`

### 6. 添加 API 端点

- `POST /api/commands/picker_uia` → 启动 `scripts/picker_uia.py`，返回捕获的 UIA 控件 JSON
- 前端 `api.js` 新增 `runPickerUia()`
- 元素库增加绿色「UIA」拾取按钮（与紫色「桌面」并列）

## 二、遇到的问题 & 怎么解决的

### 问题 1: `wintypes.WNDCLASSW` 不存在

**现象**: 创建覆盖窗口时报 `AttributeError: module 'ctypes.wintypes' has no attribute 'WNDCLASSW'`

**解决**: 手动定义 `WNDCLASSW` 结构体（13 个字段），用 `ctypes.cast()` 转换函数指针为 `c_void_p`。

### 问题 2: GDI 函数参数类型不匹配

**现象**: `SetROP2(hdc, R2_XOR)` 报 `OverflowError: int too long to convert`

**根因**: 未给 GDI 函数声明 `argtypes`/`restype`，ctypes 默认将返回值当作 Python int（可能溢出 32 位）。`GetDC` 返回的 HDC 在 64 位系统上是 64 位值，不声明类型会被截断。

**解决**: 给所有 GDI 函数声明类型：
```python
_GetDC.argtypes = [wintypes.HWND]; _GetDC.restype = wintypes.HDC
_ReleaseDC.argtypes = [wintypes.HWND, wintypes.HDC]
_SetROP2.argtypes = [wintypes.HDC, ctypes.c_int]
# ... 等 10+ 个函数
```

### 问题 3: UIA `ControlFromPoint` 导致拾取器卡死

**现象**: 在鼠标轮询循环中每次调 `uiautomation.ControlFromPoint`，拾取器卡住不响应。

**根因**: UIA 是 COM 接口，每次调用需要 `CoInitialize/CoUninitialize`，开销大。40ms 循环中反复调用 COM 初始化导致整个循环阻塞。

**解决**: 混合方案——
- **热循环**（每帧）用 Win32 `WindowFromPoint`（纯 C 函数，亚毫秒级）
- **捕获时**（仅一次）调 UIA `ControlFromPoint` + `GetParentControl` 链回溯

### 问题 4: ANSI 转义码污染 JSON 输出

**现象**: `picker_uia` 输出混入 `\u001b[?25l`（光标隐藏）导致 JSON 解析失败。

**解决**: 
- 脚本端：所有转义码写入 `stderr` 而非 `stdout`
- 后端：`commands_router.py` 用正则 `\x1b\[[0-9;?]*[a-zA-Z]` 过滤 stdout 中的转义码

### 问题 5: UIA 拾取器捕获到错误的控件

**现象**: 鼠标点在按钮上，捕获到的是 `TitleBarControl` 或 `EdgeBrowserFrameViewWin`

**根因**: UIA 控件层级比 Win32 深很多。`ControlFromPoint` 返回的是 UIA 树中该点最顶层能识别的控件。浏览器控件走 UIA 有独立 accessibility tree。

**现状**: 在标准 Win32 应用（记事本、文件对话框）上工作正常。浏览器/Electron/WPF 应用需进一步调试 UIA 树结构。

## 三、还存在的问题

1. **UIA 拾取精度**: 对浏览器和自绘控件的 UIA 树解析不完善，可能捕获到容器控件而非子控件。需要 `TreeWalker` 深度遍历或 `ElementFromPoint` 的 `hitTest`。

2. **旧指令未删除**: 原来的 12 个不带后缀的 JSON/Python 文件仍存在。旧工作流节点的 `cmd` 字段引用的是旧名，需要迁移或保持向后兼容。

3. **UIA 元素库路径查找不稳定**: `pickElementUia` 的路径匹配依赖 `name`/`class_name`/`automation_id` 三元组。`name` 和 `automation_id` 常为空，导致匹配失败。

4. **未测试完整 UIA 流程**: 端到端（拾取 → 存入元素库 → pickElementUia → inputElementUia/clickElementUia）未跑通。

5. **UIA 拾取器高亮不精确**: Win32 `WindowFromPoint` 高亮的是整个顶层窗口矩形，不是 UIA 子控件边界。改为全 UIA 高亮会卡，维持 Win32 高亮又不够精确——需要在二者间找平衡。

6. **`pip install uiautomation` 依赖**: 是第三方库，需要额外安装。优点是 API 简洁，缺点是部署多一步。
