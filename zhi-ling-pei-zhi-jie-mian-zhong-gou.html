指令配置界面重构方案：从“表单堆砌”到“向导式配置”

一、现状问题

当前 /admin/commands 的编辑弹窗把所有配置项平铺在一个长表单里，随着字段增多，配置人员容易迷失。主要痛点：

问题表现影响
信息密度过高基本信息、Runtime、判定速查、作用域说明、参数表全部挤在一个弹窗一眼看不到重点
英文字段裸露Handler 下拉是 elementAction、navigate、wait 等英文名；字段类型是 locator、varName非技术人员难理解
参数表过宽9 列挤在表格里，横向滚动，默认值/占位符/选项混为一格添加参数效率低、易填错
缺乏即时反馈保存前看不到指令卡片长什么样、参数表单长什么样反复保存-刷新-验证
辅助文档喧宾夺主Runtime 判定标准和容器作用域说明占据大块空间真正要填的输入框被淹没
没有配置向导新增指令时不知道先填什么、后填什么容易漏填关键项

二、设计目标

  1. 降低认知负荷：一次只展示一类信息，分步骤/分卡片配置。
  2. 全中文界面：系统字段（handler、字段类型）用中文标签展示，英文标识仅作 secondary 提示。
  3. 所见即所得：实时预览指令卡片和参数表单。
  4. 减少重复输入：根据指令类型和字段名自动推荐 handler、local、action 等默认值。
  5. 保持实现成本可控：复用现有 /api/commands 接口，仅改造前端模板。

三、推荐方案：四栏向导 + 实时预览

3.1 整体布局

将编辑弹窗改为横向四栏：

┌─────────────────────────────────────────────────────────────────────────────┐
│ 步骤导航 (上)                                                               │
│ [1. 基础信息] [2. 执行方式] [3. 参数] [4. 预览]                              │
├──────────────┬──────────────────────────────────────────────┬───────────────┤
│ 指令列表     │  当前步骤表单                                │  实时预览区   │
│ (可选)       │                                              │               │
│              │                                              │               │
└──────────────┴──────────────────────────────────────────────┴───────────────┘

如果空间有限，可简化为“上步骤导航 + 左表单 + 右预览”三栏。

3.2 步骤一：基础信息

只保留最核心的展示属性：

中文标签系统字段说明
指令标识type英文唯一标识，readonly（编辑时）
显示名称label左侧面板与节点上显示的中文名
所属分类category下拉选择 + 快速新增
图标icon图标选择器（FontAwesome 搜索）
图标颜色iconColor颜色选择器
背景颜色bgColor颜色选择器
排序categoryOrder / commandOrder合并为“分类排序 / 类内排序”
说明description在节点属性面板展示

改进点：

  • 图标/颜色/背景提供预设色板，而不是手动输入 class 名。
  • 右侧即时渲染指令卡片缩略图。

3.3 步骤二：执行方式

用三个可视化卡片代替 checkbox + 下拉框：

[ 浏览器执行 ]    [ 后端执行 ]    [ 无执行 ]
  扩展 handler      本地 handler     容器/结构/自定义

选择后展示二级选项：

  • 浏览器执行：
    • 动作类型：点击 / 输入 / 提取 / 滚动 / 悬停 / 导航 / 等待 / 按键 / 其他
    • 系统根据动作类型自动填充 handler（如 elementAction）和 extra.action。
    • 高级：允许手动覆盖 handler（仅开发人员）。
  • 后端执行：
    • 动作类型：设置变量 / 追加列表 / 日志 / HTTP / 自定义代码 / 其他
    • 自动映射到 @register_local 的 handler。
  • 无执行：
    • 自动勾选容器/结构标记，提示填写 closesWith。

改进点：

  • 配置人员不再看到 raw JS 函数名，只看到中文动作类型。
  • 智能检验按钮放在本步骤底部，点击后给出“推荐执行方式”的中文结论。

3.4 步骤三：参数配置

把表格改成参数卡片列表，每个参数一张卡片，支持拖拽排序：

┌─────────────────────────────────────────┐
│  元素        [×] [↑] [↓]                │
│  参数名: element_name                   │
│  显示名: 元素                            │
│  类型: 元素选择 (elementName)            │
│  必填: [●]  分组: 输入                   │
│  [展开更多]                              │
└─────────────────────────────────────────┘

每个参数的可配置项：

配置项说明
参数名系统字段，英文
显示名称中文标签
类型下拉：文本 / 数字 / 开关 / 下拉选择 / 多行文本 / 元素选择 / 变量名 / 隐藏字段
必填开关
分组输入 / 输出 / 高级
默认值根据类型渲染不同输入框
占位提示placeholder
选项仅 select 类型显示，支持“标签:值”
描述鼠标悬停提示

改进点：

  • 隐藏字段（如 action）默认折叠，避免干扰。
  • 类型选择带图标和中文说明。
  • 新增参数时根据当前指令智能推荐（如 elementAction 自动加 element_name）。

3.5 步骤四：预览与校验

右侧固定预览区展示：

  1. 指令卡片：图标、名称、分类、执行方式徽章（扩展/本地/无）。
  2. 节点属性表单预览：参数按分组渲染，验证必填、默认值是否生效。
  3. 校验结果：
    • 推荐执行方式是否匹配
    • 是否有参数缺失
    • 是否所有英文字段都有中文标签
    • handler 是否存在

3.6 列表页优化

  • 指令卡片增加“执行方式”色块徽章：蓝色=扩展，灰色=本地，橙色=容器，紫色=结构。
  • 审核模式把 handler 和 local 显示为中文标签而非英文。
  • 支持批量操作：批量启用/禁用、批量改分类排序。

四、实施建议

方案 A：最小可行改造（推荐）

只改造编辑弹窗，不改动列表页：

  1. 弹窗内增加步骤导航条。
  2. 第一步基础信息 + 实时卡片预览。
  3. 第二步执行方式改为三卡片 + 中文动作类型映射。
  4. 第三步参数表改为卡片列表。
  5. 第四步合并预览与智能校验。

优点：改动集中，一周可完成，效果立竿见影。 缺点：列表页仍较密，但编辑体验已大幅改善。

方案 B：完整重构

同时改造列表页，引入虚拟滚动、批量编辑、分类折叠记忆、搜索高亮等。

优点：整体体验统一。 缺点：工作量大，需要两周以上。
