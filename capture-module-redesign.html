捕获新元素 / 捕获子元素模块重构方案

执行摘要

将侧板的"捕获新元素"与"捕获子元素"重新划分为两个独立语义模块：

  • 捕获新元素：生成并保存全局选择器，元素类型为 plain 或 anchor。
  • 捕获子元素：必须基于已选循环锚点，生成相对选择器，元素类型固定为 child。

核心变更是在后端显式引入 WorkflowElement.element_kind ∈ {plain, anchor, child}，并以前端 activeAnchor 为唯一锚点上下文，让推荐方案与手动编辑都驱动同一份相对选择器，而不是两套并行的输入。

决策建议：先落地数据模型与保存校验，再调整前端自动重算逻辑，最后补端到端测试。整体改动集中在 extension 与 repo/service 两层，不引入新外部依赖。

1. 现状

层级已具备能力缺口
后端WorkflowElement.relative_selector、anchor_selector、anchor_element_name、anchor_mode 已存在并持久化没有字段区分元素是"锚点 / 子元素 / 普通元素"
后端elements_service.py 已解析相对字段未对 child 元素强制校验 anchor_element_name
扩展脚本content_capture.js 已维护 activeAnchor，支持 setActiveAnchor、computeRelativeFromAnchorperformCapture 只在命中锚点实例时才生成相对选择器，未与 sidepanel 的 child 模式强绑定
侧板sidepanel.js 已有 `captureMode = 'new' \'child'、activeAnchorSelect`、手动编辑标志推荐方案切换时不会自动重算相对选择器；锚点行与 anchorCard 职责重叠
类型层WorkflowElementIn/Out 已有相对字段缺少 element_kind 与 child 锚点关系约束

当前流程已能完成"先选锚点再捕获"的 happy path，但数据模型把 child 元素当成"带相对字段的普通元素"，导致导出器和执行器无法仅凭数据判断解析策略。

2. 核心问题

P1 子元素身份不明确
数据库中没有字段表明某元素是子元素。导出/执行时只能通过 `anchor_element_name` 是否为空推断，容易与历史手动填写的相对选择器混淆。
P1 推荐方案与相对选择器脱节
在 child 模式下切换全局 candidate 或 family 时，`relativeSelectorInput` 不会随之重新计算，用户必须手动点"重新计算"。
P2 手动编辑状态无法回到推荐状态
一旦 `relativeSelectorInput` 被编辑过，`anchorMode` 永远为 `manual`；切回"推荐方案"标签后没有机制重置为 `anchor-first`。
P2 activeAnchorSelect 与 anchorCard 双头管理
顶部下拉负责页面高亮和捕获上下文，anchorCard 负责已捕获元素的相对选择器编辑，二者在 child 模式下需要保持双向同步，容易漂移。
P2 anchor_mode 语义过多
当前有 `'auto'、'manual'、'none'、'backfill'、'anchor-first'`，其中 `auto` 与 `anchor-first`、`backfill` 与 `manual` 边界不清。

3. 设计目标

  1. 显式类型：后端用 element_kind 区分 plain / anchor / child。
  2. 锚点优先：child 模式下必须先选择 activeAnchor，否则禁用捕获与保存。
  3. 相对选择器为主：child 元素的主选择器是相对选择器；全局选择器仅作为后备与校验。
  4. 动态联动：推荐方案变更时自动重算相对选择器；手动编辑时标记为 manual。
  5. 来源可回退：从手动编辑切回推荐方案时，恢复 anchor_mode='anchor-first'。
  6. 后端强校验：保存 child 元素时强制要求 anchor_element_name 且能在同流程找到对应 anchor。
  7. 导出语义清晰：运行时按 element_kind 路由，child 元素通过锚点元素名称解析到锚点选择器。

5. 前端模块设计

5.1 捕获新元素（captureMode='new'）

  • 仅生成全局候选列表（CSS / XPath）。
  • 隐藏 activeAnchorRow 与 anchorCard。
  • 保存时：
    • 默认 element_kind='plain'。
    • 用户可勾选"此元素可作为循环锚点"，则 element_kind='anchor'。
  • 离开 child 模式进入 new 模式时，清除 activeAnchor 与页面高亮。

5.2 捕获子元素（captureMode='child'）

模块边界：child 模式 = 选择锚点 + 捕获子元素 + 相对选择器管理 + 校验。所有推荐/手动编辑都围绕 activeAnchor 展开。

5.2.1 锚点选择

  • 强制显示 activeAnchorRow。
  • 下拉选项仅包含当前流程中 element_kind='anchor' 的元素；若流程内无 anchor 元素，显示提示"请先捕获一个循环锚点元素"。
  • 未选择锚点时：禁用"捕获"快捷键提示、禁用"保存"按钮。
  • 选择锚点后：向 content_capture.js 发送 setActiveAnchor，页面持续高亮所有锚点实例。

5.2.2 捕获行为

performCapture 在 child 模式下：

  1. 检查 activeAnchor 是否存在，不存在则提示并拒绝捕获。
  2. 找到包含被点击元素的锚点实例 host。
  3. 调用 buildRelativeCss(host, el)，失败再试 buildRelativeXPath。
  4. 生成的 payload 中：
    • relativeSelector = 相对选择器（必填）
    • anchorSelector = 锚点全局选择器
    • anchorElementName = 锚点元素名称
    • anchorMode = 'anchor-first'
    • elementKind = 'child'
  5. 候选列表排序：优先展示以该锚点为根的相对路径候选；全局候选折叠在"全局后备"分组中。

5.2.3 推荐方案联动

  • 选择不同 candidate 或切换 CSS/XPath family 时，自动调用 computeRelativeFromAnchor 更新 relativeSelectorInput。
  • 更新后重置 relativeManuallyEdited = false，anchorModeLabel 显示"锚定"（对应 anchor-first）。

5.2.4 手动编辑

  • "手动编辑"模式下 relativeSelectorInput 可编辑。
  • 输入任意字符即设置 relativeManuallyEdited = true，anchorModeLabel 显示"手动"。
  • 校验按钮调用 verifyRelative，统计每个锚点实例内的匹配数。
  • 切回"推荐方案"时：
    • 如果当前 relative selector 与推荐生成的相同，保持 manual 不降级？否：只要回到 recommend，就按推荐来源重算并恢复 anchor-first，除非用户明确点"锁定手动值"。

5.2.5 activeAnchorSelect 与 anchorCard 同步规则

场景activeAnchorSelect（顶部）anchorCard（相对选择器卡片）
用户选择锚点更新选中项，推送页面高亮anchorSelectorInput 同步为锚点全局选择器，relativeSelectorInput 清空或重算
捕获完成回显保持选中（如果 payload 的 anchorElementName 一致）填充相对/锚点字段
用户编辑 anchorSelectorInput不反向修改 activeAnchorSelect；保存时以 anchorCard 为准允许编辑，但校验失败时禁止保存
切换 captureMode 到 new清空并发送 setActiveAnchor(null)隐藏

6. 后端服务变更

6.1 src/service/elements_service.py

def save_captured_element(payload: dict, workflow_id: int) -> WorkflowElement:
    kind = payload.get("elementKind", "plain")
    if kind not in {"plain", "anchor", "child"}:
        raise ValueError("invalid element_kind")

    if kind == "child":
        anchor_name = payload.get("anchorElementName") or payload.get("anchor_element_name")
        if not anchor_name:
            raise ValueError("child element requires anchor_element_name")
        anchor_el = lookup_anchor_by_name(workflow_id, anchor_name)
        if not anchor_el or anchor_el.element_kind != "anchor":
            raise ValueError("anchor not found or not an anchor element")

    relative = (payload.get("relativeSelector") or "").strip()
    manually_edited = payload.get("relativeManuallyEdited", False)
    anchor_mode = "manual" if manually_edited else ("anchor-first" if kind == "child" else "none")

    element = WorkflowElement(
        workflow_id=workflow_id,
        name=payload["name"],
        element_kind=kind,
        selector=payload.get("selector", ""),
        selector_family=payload.get("selectorFamily", "css"),
        relative_selector=relative,
        anchor_selector=payload.get("anchorSelector", ""),
        anchor_element_name=anchor_name if kind == "child" else None,
        anchor_mode=anchor_mode,
        # ... 其他字段
    )
    # 如果 kind == 'anchor' 且用户未提供全局选择器，可拒绝保存
    return element

6.2 导出器 / 运行时

  • element_kind='anchor'：按全局选择器执行，循环迭代时每个实例作为上下文根。
  • element_kind='child'：在同流程查找 anchor_element_name 对应的 anchor 元素，使用其选择器作为上下文根，再用 relative_selector 解析目标。
  • element_kind='plain'：保持现有全局执行逻辑。

10. 下一步决策

方案 A（推荐）：完整落地
新增 element_kind，明确 anchor/child/plain 语义，同时修复推荐/手动联动。一次性解决身份识别与交互问题，但改动面较大。
方案 B：最小改动
不新增 element_kind，仅通过 anchor_mode 区分（如 child 强制 anchor-first/manual）。改动小，但语义仍不清晰，未来重构成本更高。

推荐先实施方案 A 的数据模型层与保存校验，再逐步迁移前端自动重算逻辑，可以降低一次性改动风险。
