子元素展示与使用逻辑设计方案

1. 背景与现状

元素库当前有三种 element_kind：

  • plain：独立元素，使用 web_selector 在全局页面内匹配。
  • anchor：可作为循环锚点的元素，同样使用 web_selector 在全局页面内匹配，用于 forEachElement 循环。
  • child：依附于某个 anchor（或其他 child）的子元素，保存 relative_selector 与 anchor_element_name，在循环上下文内相对当前外层元素解析。

当前问题：

  • child 的全局 web_selector 已不再保存（空值），但后端/前端尚未明确其展示与使用约定。
  • 用户在不同场景下使用 child 时行为不一致：脱离循环时如何匹配？在循环内如何匹配？需要统一约定。
  • child 元素本身也可以作为其他 child 的锚点，形成多级相对链。

2. 树结构展示设计

2.1 数据模型层

在 WorkflowElement 输出模型中增加只读派生字段：

{
  "id": 101,
  "name": "comment_card",
  "element_kind": "anchor",
  "parent_name": null,
  "children": ["author_name"]
}
{
  "id": 102,
  "name": "author_name",
  "element_kind": "child",
  "parent_name": "comment_card",
  "children": ["author_avatar"]
}
  • parent_name：等于 anchor_element_name；若为空则为根节点。
  • children：列出所有以该元素为 anchor_element_name 的子元素名称列表。

2.2 后端构造规则

后端在返回元素列表时，按以下规则构造树：

  1. 以 plain / anchor 元素为根节点。
  2. 将 child 元素挂载到其 anchor_element_name 对应的父元素下。
  3. 若 child 的 anchor_element_name 不存在，则标记为 游离子元素，单独列出或返回警告。
  4. 树深度不限；child 也可以作为其他 child 的父节点。

2.3 API 建议

新增/扩展现有接口：

GET /api/workflows/{wf_id}/elements/tree

返回：

{
  "roots": [
    {
      "name": "comment_card",
      "element_kind": "anchor",
      "web_selector": "css:.comment-card",
      "children": [
        {
          "name": "author_name",
          "element_kind": "child",
          "relative_selector": "css:.author",
          "anchor_element_name": "comment_card",
          "children": [
            {
              "name": "author_avatar",
              "element_kind": "child",
              "relative_selector": "css:.avatar",
              "anchor_element_name": "author_name"
            }
          ]
        }
      ]
    }
  ],
  "orphans": []
}

现有 GET /api/workflows/{wf_id}/elements 保持不变，仅补充 parent_name / children 字段即可。

3. 有效选择器链（Effective Selector Chain）

由于 child 的 web_selector 为空，当需要把它当作锚点使用时，后端必须动态计算其有效选择器链：

[root.web_selector] + [child_1.relative_selector] + ... + [child_n.relative_selector]

计算规则：

  1. 从目标元素出发，沿 anchor_element_name 向上回溯，直到遇到 plain 或 anchor 元素（根）。
  2. 若任意中间节点缺失或不是有效元素，则链断裂，返回错误。
  3. 链中的每个节点必须有非空的相对选择器（根节点除外，根使用 web_selector）。

示例：

元素kindweb_selectorrelative_selectoranchor_element_name
comment_cardanchorcss:.comment-card--
author_namechild空css:.authorcomment_card
author_avatarchild空css:.avatarauthor_name

author_avatar 的有效选择器链：

  • CSS 形式：css:.comment-card .author .avatar
  • XPath 形式：xpath://*[contains(@class,'comment-card')]//*[contains(@class,'author')]//*[contains(@class,'avatar')]（需按家族分别拼接）

后端应提供接口或工具函数计算任意元素的有效选择器链：

GET /api/workflows/{wf_id}/elements/{name}/chain

返回：

{
  "name": "author_avatar",
  "chain": [
    { "name": "comment_card", "selector": "css:.comment-card", "kind": "anchor" },
    { "name": "author_name", "selector": "css:.author", "kind": "child" },
    { "name": "author_avatar", "selector": "css:.avatar", "kind": "child" }
  ],
  "combined_css": ".comment-card .author .avatar",
  "combined_xpath": "//*[contains(@class,'comment-card')]//*[contains(@class,'author')]//*[contains(@class,'avatar')]"
}

4. 使用约定：两种场景

4.1 场景 A：全页面匹配（脱离循环 / 无上下文）

定义：child 元素被用在 forEachElement 循环之外，或当前执行栈中没有匹配的锚点循环。

约定：

  • 默认行为：报错，不允许直接使用。
  • 可选行为（显式降级）：如果命令的 extra.scope == "global" 或 extra.useRelative == false，则按 plain 元素处理，使用有效选择器链的全局拼接结果匹配。
    • 该降级实际上等价于“用全局链式选择器查找”，可用于调试或简单场景。

执行语义：

if element_kind == "child" and 当前无匹配锚点循环:
    if 命令显式要求全页面匹配:
        使用有效选择器链的全局形式在 document 内查找
    else:
        raise ValueError("子元素必须在 forEachElement 循环内使用")

校验规则：

  • 保存工作流节点时，后端/扩展运行器检查：若节点引用 child 元素，则默认必须位于对应 anchor_element_name 的 forEachElement 循环体内。
  • 若允许全页面匹配，需验证有效选择器链可成功计算。

4.2 场景 B：在当前外层元素内查找（循环上下文内）

定义：child 元素位于其 anchor_element_name 对应的 forEachElement 循环体内。

约定：

  • 必须使用相对解析：extra.useRelative = true。
  • 解析步骤：
    1. 找到当前循环项（外层元素）。
    2. 在该循环项内部使用 relative_selector 查找子元素。
    3. 返回匹配结果。

嵌套场景：

若 author_avatar 是 author_name 的子元素，author_name 又是 comment_card 的子元素，则工作流中需要嵌套循环：

forEachElement(comment_card)
  forEachElement(author_name)
    click(author_avatar)
  endFor
endFor

扩展运行器通过循环栈 _LOOP_STACK 匹配锚点：

stack = [comment_card, author_name]
author_avatar.anchor_element_name == author_name  -> 匹配成功

执行语义（扩展协议）：

{
  "cmdType": "click",
  "locator": "",
  "selectorFamily": "css",
  "extra": {
    "useRelative": true,
    "relativeLocator": ".avatar",
    "relativeSelectorFamily": "css"
  }
}
  • locator 对 child 元素可为空，运行器优先使用 extra.relativeLocator。
  • 不需要在 extra 中传递完整祖先链，因为当前循环项已经是直接父元素。

校验规则：

  • 保存 child 元素时，必须存在对应的 anchor_element_name 且该锚点已存在。
  • relative_selector 不能为空。
  • 推荐在捕获时校验 relative_selector 在每个父元素实例内至少命中一次。

5. 数据模型与 API 变更建议

5.1 后端模型

WorkflowElement 已包含所需字段，无需新增列。补充：

  • parent_name（只读，派生）
  • children（只读，派生）
  • 服务层工具函数 compute_selector_chain(element_name) 用于计算有效选择器链。

5.2 扩展协议

扩展执行步骤的 extra 中保留以下字段：

  • useRelative（bool，默认 true）
  • relativeLocator（去前缀后的相对选择器）
  • relativeSelectorFamily（css / xpath / drission）

对于 child 元素：

  • locator 字段保持为空，避免扩展端误用全局选择器。
  • 扩展端必须检查：若 useRelative == true 且 relativeLocator 存在，则优先使用相对解析。
  • 若需要全页面匹配，则使用后端计算好的链式全局选择器填充 locator。

5.3 捕获侧板

  • 锚点下拉框允许选择任意元素（包括 child）。
  • 当选择 child 作为锚点时，侧板调用后端 /elements/{name}/chain 计算有效选择器链，再发送给内容脚本定位锚点。
  • 若链断裂，提示用户“该子元素缺少可解析的父锚点链”。

6. 前端展示建议

6.1 元素库树形组件

  • 根节点为 plain / anchor。
  • child 可无限嵌套，按 anchor_element_name 挂载。
  • 游离子元素（anchor_element_name 不存在）用警告色标注。
  • 点击任意节点可查看其有效选择器链。

6.2 节点编辑器

  • 当节点引用的元素为 child 时：
    • 默认强制 使用相对解析。
    • 允许显式切换为“全页面匹配”，此时使用有效选择器链的全局形式。
    • 显示所属父元素链（breadcrumb）。
